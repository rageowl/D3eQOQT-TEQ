# 작업 지시: 인터랙티브 매뉴얼 구현

본 문서는 Claude Code가 **이 저장소의 소스코드를 직접 분석하고** `MANUAL_SPEC.md`의 결과물을 구현하기 위한 작업 절차다.

먼저 `MANUAL_SPEC.md`를 끝까지 읽고 시작할 것.

---

## 작업 원칙

1. **추측 금지**: UI 동작이나 기능을 추측해서 매뉴얼을 쓰지 않는다. 반드시 소스코드에서 근거를 찾는다.
2. **체크포인트 저장**: 각 페이즈가 끝날 때마다 `_help_progress.json`에 진행 상태를 기록한다. 세션이 끊겨도 이어서 작업할 수 있어야 한다.
3. **기존 코드 비침해**: `MANUAL_SPEC.md` §3.7 원칙을 따른다.
4. **언어**: 모든 매뉴얼 콘텐츠는 한국어 + 일본어 둘 다. 한국어를 먼저 쓰고, 일본어는 한국어 번역으로 채운다.

---

## Phase 1: 소스코드 인벤토리

목표: 어떤 파일에 무엇이 있는지 전체 그림을 그린다.

### 1.1 파일 트리 작성

저장소 루트의 모든 HTML/JS/CSS 파일을 나열하고, 각 파일의 역할을 한 줄로 요약한다. 결과를 `_help_inventory.md`에 저장:

```markdown
# Source Inventory

## HTML
- `index.html` — 메인 페이지, 모든 UI 요소의 DOM 정의

## JS
- `app.js` — 진입점, 초기화 로직
- `player.js` — 유튜브 임베드 제어, 재생 로직
- ...
```

### 1.2 UI 요소 ↔ 소스 매핑

`MANUAL_SPEC.md` §2.1의 10개 영역 각각에 대해:

- 해당 UI를 정의하는 HTML 위치 (파일명 + 라인 또는 셀렉터)
- 동작을 구현하는 JS 함수 위치
- 발견된 단축키 / 우클릭 메뉴 / 숨겨진 기능

이를 `_help_mapping.md`에 표로 정리:

```markdown
| Section ID | UI 위치 (HTML) | 동작 구현 (JS) | 비고 |
|---|---|---|---|
| topbar-data-toggles | index.html `header > div.toggles` | app.js:onToggleLibrary() 등 | ... |
```

### 1.3 누락 영역 발견 보고

스펙의 10개 영역 외에 매뉴얼이 필요한 추가 UI/기능이 발견되면 별도 섹션으로 보고한다. 사용자가 승인하면 §2.1에 추가한다.

**Phase 1 완료 조건**: `_help_inventory.md`, `_help_mapping.md` 작성 완료, 사용자에게 누락 영역 검토 요청.

## Phase 2: 콘텐츠 작성

목표: `help/help-content.js`에 들어갈 모든 텍스트를 확정한다.

### 2.1 섹션별 콘텐츠 초안 (한국어)

Phase 1의 매핑을 근거로 각 섹션에 대해 다음을 작성:

- `title.kr`: 영역 이름 (영문 원문 + 한글 의미, 예: `Batch Edit (일괄 편집)`)
- `short.kr`: 30자 이내 한 줄 설명 (말풍선용)
- `detail.kr`: 마크다운 본문. 다음 구조를 따른다:

```markdown
### 무엇인가요
한 문단으로 기능 설명.

### 어떻게 쓰나요
1. 단계별 사용법
2. ...

### 이럴 때 씁니다
- 구체적 사용 예시 1
- 구체적 사용 예시 2

### 함께 보기
- 관련 섹션으로의 링크 (related 필드와 연동)
```

### 2.2 개념 섹션 작성 (한국어)

`MANUAL_SPEC.md` §2.2의 4개 개념 섹션을 동일한 구조로 작성한다. 특히 "Library와 PlayList의 차이"는 **다이어그램**(ASCII 또는 SVG)을 포함하면 좋다.

### 2.3 일본어 번역

한국어 확정본을 일본어로 번역. 번역 시 주의:

- UI 버튼 이름은 영문 그대로 유지 (`Library`, `PlayList` 등)
- 일본어 존댓말(です/ます체) 사용
- 한국어 원문에 없는 정보를 임의로 추가하지 않음

### 2.4 콘텐츠 검토 체크리스트

작성된 모든 텍스트에 대해 자체 검수:

- [ ] 추측이 아닌 소스코드에서 확인한 사실인가
- [ ] 영문 버튼 이름이 정확히 일치하는가 (대소문자 포함)
- [ ] 한 문장이 너무 길지 않은가
- [ ] "이럴 때 씁니다" 예시가 구체적인가
- [ ] 한국어/일본어 의미가 일치하는가

**Phase 2 완료 조건**: `help/help-content.js` 작성 완료. 사용자에게 텍스트 검토 요청 (특히 일본어).

## Phase 3: 셀렉터 안정화

목표: 각 섹션의 `selector`가 안정적으로 대상 요소를 가리키도록 한다.

### 3.1 셀렉터 후보 작성

Phase 1.2 매핑을 기반으로 각 섹션의 1차 셀렉터를 작성한다. 우선순위:

1. 기존 ID (`#some-id`)
2. 의미 있는 단일 클래스 (`.player-controls`)
3. 복합 셀렉터 (`header > div.toggles`)

### 3.2 `data-help` 속성 추가

복합 셀렉터에 의존하는 영역은 안정성을 위해 원본 HTML에 `data-help="<section-id>"` 속성을 추가한다.

**규칙**:
- 추가만 한다. 기존 ID/클래스/이벤트 핸들러는 절대 수정하지 않는다
- 추가한 속성 목록을 `_help_data_attrs.md`에 기록 (롤백 가능하도록)
- 한 번에 다 바꾸지 말고, 섹션 하나씩 추가 → 페이지 정상 동작 확인 → 다음 섹션

### 3.3 셀렉터 검증

브라우저 콘솔에서 각 셀렉터를 `document.querySelector(...)`로 실행해 단일 요소가 정확히 매치되는지 확인. 검증 결과를 `_help_selector_check.md`에 기록.

**Phase 3 완료 조건**: 모든 섹션의 셀렉터가 검증됨. 추가된 `data-help` 속성 목록이 문서화됨.

## Phase 4: 헬프 모드 구현

목표: `MANUAL_SPEC.md` §3의 기술 요구사항대로 동작하는 헬프 모드를 구현한다.

### 4.1 파일 생성

`help/` 디렉토리에 다음 4개 파일 생성:

- `help-content.js` — Phase 2의 결과물
- `help-mode.css` — 오버레이/마커/패널 스타일
- `help-panel.js` — 사이드 패널 컴포넌트
- `help-mode.js` — 진입 로직, 마커 렌더링, 키보드 핸들러

`index.html`에 §3.2의 `<link>`/`<script>` 4줄 추가.

### 4.2 구현 순서

작은 단위로 동작 확인하며 진행:

1. **최소 동작**: `?` 버튼 → 오버레이 표시 → `Esc`로 닫기
2. **마커 표시**: 모든 섹션의 셀렉터 위에 번호 마커 부착
3. **말풍선**: 마커 클릭 시 `short` 텍스트 표시 + 다음/이전/닫기 버튼
4. **사이드 패널**: `자세히` 버튼 → `detail` 마크다운 렌더링
5. **언어 토글**: KR/JP 전환
6. **자유 탐색 모드**: 아무 마커나 클릭 가능
7. **URL 해시**: `#help`로 진입
8. **키보드 네비게이션**: `Tab`, `←/→`, `Esc`
9. **접근성 속성**: `aria-*`, 포커스 관리

### 4.3 driver.js 통합 판단

§3.1에 따라 driver.js의 spotlight 기능만 쓰고 나머지는 자체 구현. 단, 자체 구현이 driver.js의 기본 UX와 자연스럽게 통합되지 않으면 **driver.js를 빼고 모두 자체 구현하는 것도 허용**한다. 결정 근거를 코드 주석에 남길 것.

### 4.4 수동 테스트

`MANUAL_SPEC.md` §4의 수락 기준 8항목을 실제 브라우저에서 모두 검증. 결과를 `_help_test_report.md`에 기록.

**Phase 4 완료 조건**: 수락 기준 모두 통과. 사용자에게 시연 가능한 상태.

## Phase 5: 마무리

### 5.1 회귀 테스트

헬프 모드 OFF 상태에서 기존 기능이 100% 동일하게 동작하는지 확인:

- Import / Export
- 곡 재생, 이전/다음
- 정렬, 검색, 필터
- Batch Edit
- Shuffle, PlayAll

### 5.2 임시 파일 정리

`_help_*.md` 같은 작업 산출물을 어떻게 처리할지 사용자에게 확인:
- (a) 저장소에 `docs/help-build/` 형태로 보존
- (b) `.gitignore`에 추가
- (c) 삭제

### 5.3 README 업데이트

저장소에 `README.md`가 있다면 헬프 모드 사용법을 한 단락 추가:

```markdown
## Help Mode

페이지 우측 상단의 `?` 버튼을 클릭하면 인터랙티브 매뉴얼이 활성화됩니다.
KR/JP 언어 전환을 지원하며, `#help` URL로 직접 진입할 수도 있습니다.
```
